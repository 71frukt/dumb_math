# Реализация и тестирование функции ln(x)

Проект представляет собой интерфейсную библиотеку с реализованной функцией расчёта логарифма пониженной точности ($< 3.0$ ULP для $\texttt{float}$ и $< 5.0$ ULP для $\texttt{double}$ ). Функция поддерживает *Floating point flags* и *errno*.
Функция реализует следующий интерфейс:
```cpp
template <std::floating_point T, size_t TaylorDeg = sizeof(T)>
T dumb_math::logarithm::ln(T x)
```
Функция использует метод lookup-таблицы размером $2^8 = 256$ строк и полиномиальную аппроксимацию рядом Тейлора степени `TaylorDeg`. Эмпирически под требования к функции подобрано соотношение `TaylorDeg = sizeof(T)`.

### Общие сведения
Алгоритм генерации произвольной математической функции состоит из нескольких этапов:
1. Фильтрация аргумента
2. Редукция аргумента к интервалу, удобному для аппроксимации
3. Аппроксимация
    * Путь 1
        * Lookup table
        * Полиномиальная аппроксимация
    * Путь 2
        * Начальное приближение
        * Итеративные методы
4. Реконструкция результата

Погрешность алгоритма оценивается в единицах *ULP (unit in the last place)* &mdash; расстояние между двумя соседними представимыми числами, ближайшими к данному.

Дробные числа *T* $x$ с мантиссой в $\mathtt{\mathtt{mantissa\\_len}}$ бит с одинаковой экспонентой $\mathtt{exp}(x)$ распределяются вдоль одного отрезка длиной $2^{\mathtt{exp}(x)}$.  
Количество этих чисел равно $2^{\mathtt{mantissa\\_len}}$.  
Тогда $ulp(x) = 2^{\mathtt{exp}(x) - \mathtt{mantissa\\_width}}$

Если эталонная функция &mdash; $f(x)$, а тестируемая &mdash; $g(x)$, то ошибка равна 

$$ulp_{error}(x) = \frac{|f(x) - g(x)|}{ulp(x)}$$

## Описание алгоритма расчёта натурального логарифма
## Первая итерация 

### Фильтрация
&emsp; Область определения &mdash; $E = [0, +\infty)$.  
&emsp; $ln(0) = -\infty$;  
&emsp; $ln(+\infty) = +\infty$  
&emsp; На других точках $E$ возвращается не-NaN значение.  
&emsp; На точках вне $E$ возвращается NaN.

### Редукция

Сначала нужно привести аргумент $x$ к каноническому виду.

* Нормализованное положительное $x$:  
$x$ = $m \cdot 2^{\mathtt{exp}}$, $m \in [1.0, 2.0)$ &mdash; нормализованная мантисса, $\mathtt{exp}$ &mdash; целочисленная экспонента.

* Денормализованное положительное $x$:  
$x = 0.000...1xx \cdot 2^{\mathtt{exp}}$ &mdash; нормализуем мантиссу.  
Ищем первую значащую единицу после точки, сдвигаем биты мантиссы влево на $t$ бит, чтобы эта единица встала перед точкой. Т.о. мы умножили $x$ на $2^t$, поэтому делим $x$ на $t$: $\mathtt{exp}$ -= $t$. 

Функции вычисления мантиссы, нормализованной мантиссы и экспоненты числа *T* $x$ определены в 
[*[get_floating_parts.hpp]*](source/dumb_math/include/dumb_math/common/get_floating_parts.hpp).

Итак, $x$ = $m \cdot 2^{\mathtt{exp}}$.  
$\ln(x) = \ln(m \cdot 2^{\mathtt{exp}}) = \ln(m) + \mathtt{exp} \cdot \ln(2), m \in [1.0, 2.0)$  

### Аппроксимация

* #### Построение lookup-таблицы

$m \in [1.0, 2.0)$ &mdash; слишком много для аппроксимации, поэтому отрезок разбивается на $2^k$ подотрезков, в центре каждого из которых ставим опорную точку $x_i$. Достав $k$ ведущих бит из $m$, определяем номер $i$ отрезка, в котором лежит мантисса. Редуцированный аргумент $v$:  
$v = m \cdot \frac{1}{x_i} = 1 + r$  
$r = m \cdot \frac{1}{x_i} - 1 \in [-\Delta, \Delta]$  
$\Delta = \frac{2^{-k}}{2} = 2^{-k-1}$, то есть $r$ тем ближе к нулю, чем больше $k$.

$$\ln(m) = \ln\left(v \cdot x_i\right) = \ln(x_i) + \ln(1 + r)$$

$$\ln(x) = \mathtt{exp} \cdot \ln(2) + \ln(x_i) + poly\_approx(r)$$

Таким образом, для каждого $x_i$ таблица должна хранить точные значения $\frac{1}{x_i}$ и $\ln(x_i)$, и расчёт логарифма произвольного числа $x$ сведется к рассчёту $poly\_approx(r)$.

Сами точные $\frac{1}{x_i}$ и $\ln(x_i)$ рассчитываются единожды при создании библиотеки.
[*[lookup_table_builder]*](source/dumb_math/modules/logarithm/tools/lookup_table_builder)

$\ln(x)$ для $x_i \in [1; 2)$ рассчитывается через ряд ареатангенса:

$$x = \frac{1 + z}{1 - z} \implies z = \frac{x - 1}{x + 1} \in [0, \frac{1}{3})$$
$$\ln\\_Arhtan(x) = 2 \sum_{k=0}^{\infty} \frac{1}{2k+1} \left( \frac{x-1}{x+1} \right)^{2k+1}$$
Аргумент лежит на меньшем полуинтервале, поэтому ряд сходится значительно быстрее, за что приходится платить операциями деления.  
Алгоритмически ряд считается до тех пор, пока следующий член не перестанет влиять на сумму. Фактически это означает, что член ряда по значению меньше ULP, сумма ряда имеет максимальную для этого метода точность.
[*[approx.hpp - LnArtgTaylorBestAcc_0to2(T x)]*](source/dumb_math/modules/logarithm/include/logarithm/common/approx.hpp)

* #### Рассчёт полиномиальной аппроксимации
Полином рассчитывается через ряд Тейлора порядка, зависящего от точности типа *T*. В коэффициентах ряда Тейлора также есть константные дроби, которые предрассчитываются в compile-time. [*[tailor.hpp]*](source/dumb_math/modules/logarithm/include/logarithm//tailor.hpp)

* #### Катастрофическая потеря точности в окрестности 1
В окрестности единицы происходит вычитание между табличным значением $\ln(x_i)$ и суммой ряда Тейлора, которые крайне близки по значению, что, как известно, приводит к катастрофической потере точности, поэтому в окрестности 1 расчёт ведется с помощью ряда ареатангенса определенного порядка.
[*[approx.hpp -  LnArtgTaylor_0to2(T x))]*](source/dumb_math/modules/logarithm/include/logarithm/common/approx.hpp)


### Реконструкция результата
$$\ln(x) = \mathtt{exp} \cdot \ln(2) + \ln(x_i) + LnTaylor(1 + r), |x - 1| > 0.15$$
$$\ln(x) = LnArtanh(x), |x - 1| <= 0.15$$

## Вторая итерация
Недостатки первого способа заметны и заключаются в том, что во избежание катастрофической потери точности для аргумента $x$ в окрестности $1$ приходится вести расчёт отдельно через ряд ареатангенса, осуществляя многочисленные операции деления. Это приводит к замедлению программы в целом. 

Решение - построить lookup-таблицу так, чтобы она хранила опорную точку $x_0 = 1$, $ln(x_0) = 0.0$.

Это можно осуществить, выбирая $x_i$ на местах стыков прошлых отрезков. Отрезки сместим на $\frac{\delta}{2}$ влево, где $\delta = x_{i+1} - x_i$, тогда $x_i$ останутся же серединами отрезков, но первый отрезок будет $[1-\frac{\delta}{2}, 1+\frac{\delta}{2}]$, имея виртуальную часть $[1-\frac{\delta}{2},1)$, а $x_0 = 1.0$. В этом случае изменится алгоритм выбора индекса в таблице для мантиссы $m \in [1; 2)$: раньше мы производили усечение до $k$ значащих бит ($2^k$ - количество интервалов, и, соответственно, середин интервалов прошлой таблицы); сейчас на месте стыков прошлых интервалов стоят новые $x_i$, то есть размер новой таблицы - $2^k+1$, а индекс $i$ определяется округлением $m$ к ближайшему $x_i$. 

Проблема катастрофической потери точности снова возникает, но для $m$ вблизи $2$: там получим $x_i = 2$, $ln(x_i) = \ln\\_Arhtan(2)$. Расчёт через ряд ареатангенса имеет малую, но всё же погрешность, из-за чего разность приводит к большой ошибке. Во избежание этого значение таблицы для $x_i = 2$ прописывается отдельно точным значением.  

Отдельная "снайперская" ветка расчёта через ареатангенс для $|x - 1| <= 0.15$ не нужна.

Эта точность избыточна, поэтому уменьшим степень полинома Тейлора до $\texttt{sizeof(T)/4}$, то есть $1$ для $\texttt{float}$ и $2$ для $\texttt{double}$ на моей машине. Элемент таблицы состоит из двух чисел типа $\texttt{long double}$, имеющих размер 16 байт на моей машине. Вся таблица имеет размер $16 \times 257 = 4112$ байт. $\texttt{L1}$ кэш на моём компьютере имеет размер по 48 КБ на ядро, то есть таблица составляет $8\%$.
